# R1 Network_Bridge vs 当前矿场 zrok 公网桥：架构考古对比

> 依据：`r1-src/Network_Bridge.ps1`（R1 时代网络桥脚本，38 行级别的“桥之宣言”）  
> 对比对象：当前矿场的公网桥（`zrok reserved share` + `Cloudflare Worker`，以 `https://data-r1.shares.zrok.io` 这类公网入口形态为代表）

## R1 桥的设计哲学

R1 的桥本质是“内网穿透 + 局域网对齐”，目标不是把服务暴露给全网，而是让 **公司内网(Intranet)** 和 **本地网络(Local)** 之间形成一条可控的贯通通道。

从脚本结构可以读出三个核心动作：

1. **局域网身份识别**：通过 `Get-NetIPAddress` 选取 Wi‑Fi/Ethernet 的 IPv4 地址并打印 `Local Identity`，先回答“我是谁、我在哪个网段”。
2. **穿透/端口对齐（口径很像网关）**：`Setup-Penetration` 里写死了一个“网关式映射”叙事：`Local Port 3000 (Gateway) <-> LAN Port 8080`。它把“穿透”描述成端口口径对齐，而不是反向代理或公开域名。
3. **广播宣告（主权宣言）**：`Broadcast-Presence` 明确写着“向公司内网宣告 R1 主权”，并尝试执行 `05_LINKS\win_broadcast.ps1`。这意味着 R1 的可达性不仅靠连接，更靠 **在内网里被发现**（discovery / presence）。

## 当前 zrok 公网桥的设计哲学（reserved share + Worker）

当前矿场桥更像“公网暴露入口”，核心目的是让外部环境能稳定访问矿场能力或数据接口：

1. **公网地址即能力**：通过 `zrok reserved share` 把某个服务映射到一个稳定的公网子域（例如 `data-r1.shares.zrok.io`），入口天然适配外部调用与跨网络访问。
2. **零配置/轻运维倾向**：更少依赖内网广播、网段识别、局域网协议；更强调“拿到 URL 就能用”的体验。
3. **代理模式而非端口对齐叙事**：对外呈现为 HTTP/HTTPS 入口（通常还会叠加 `Cloudflare Worker` 做转发、路由、鉴权/限流、缓存或统一 Header 处理等），整体更接近“边缘代理/网关”。

## 核心差异（古今桥的世界观冲突点）

| 维度 | R1 Network Bridge | 当前矿场 zrok 公网桥 |
|---|---|---|
| 主要战场 | **内网穿透**：公司 Intranet + 本地网 | **公网暴露**：任何网络都可访问 |
| 可达性机制 | **主动广播**：宣告存在、被内网发现 | **被动代理**：外部按 URL 来访问 |
| 身份叙事 | “先认网段/身份，再建立隧道” | “入口域名就是身份（可叠加 Worker 做规则）” |
| 安全边界默认值 | 默认在内网语境里“可见但可控” | 默认在公网语境里“可见且必须防护” |
| 运维依赖 | 依赖环境脚本与内网机制（如 `win_broadcast.ps1`） | 依赖隧道/边缘层与 DNS/域名（+ Worker） |

一句话总结：**R1 是把“内网”当主宇宙，把公网当例外；我们是把“公网”当主宇宙，用边缘层把风险关起来。**

## 对矿场的启示：哪些思路值得吸收

1. **补一条“内网态”能力：可选的 LAN discovery**
   - 现在公网桥解决了“远程可达”，但局域网里“自动发现矿场节点/服务列表”这件事可能仍缺位。
   - 可以借鉴 R1 的“广播宣告”理念：在可信内网里，用轻量广播/注册（mDNS、UDP 广播、或一个内网 registry）让同网段节点能快速互相发现。

2. **把“Local Identity”变成可观测指标**
   - R1 先打印本机 IPv4，本质是在做“我在哪个网络平面”的可观测性。
   - 矿场可以把当前节点的网络态（内网 IP、公网出口、当前 share URL、Worker 路由版本）纳入健康检查与 dashboard 指标，降低排障成本。

3. **端口对齐叙事有助于清晰边界**
   - R1 把桥讲成“3000 网关 <-> 8080 内网服务”，这种口径能强迫大家明确：谁是入口、谁是内网服务、谁负责鉴权。
   - 现在公网桥（zrok + Worker）也建议固定叙事口径：Worker 是“公网网关”，源站/服务是“内网服务”，把职责写清楚会更利于演进。
